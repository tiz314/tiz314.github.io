For a recent network security course during my master's, I developed a practical lab project exploring the security implications of modern aerospace architectures. As satellite ground station networks increasingly transition to edge-deployed microservices and Kubernetes, they inherit the exact same vulnerabilities found in standard cloud environments.

This project demonstrates how a threat actor can chain together software vulnerabilities and container misconfigurations to move from a remote edge node all the way to complete Host OS compromise. Since the full academic report isn't public, here is a quick breakdown of the rationale, the setup, and the multi-stage kill chain.

### The Setup: Edge Meets the Control Room
The scenario models a simplified, geographically distributed ground station network. To make this accessible for lab environments, everything is provisioned on a Kubernetes instance simulating two primary components:

* **The Ground Station (Edge):** A Python-based pod acting as a mockup for a telemetry signal parser and an HTTP server.
* **The Control Room (Central):** A pod running an older version of Logstash (6.3.0), which periodically polls the ground station for JSON telemetry data.

Because these niche aerospace deployments often rely on standard Commercial-Off-The-Shelf (COTS) software, the attack surface is surprisingly familiar.

### The Kill Chain: A Three-Stage Ascent
To gain root access to the host operating system, the attacker executes a three-stage exploit chain.

**1. Edge Compromise (Signal Parser RCE)**
The attack starts at the edge. The ground station uses a mock signal parser that evaluates decoded telemetry files without proper sanitization. By feeding a spoofed telemetry payload containing malicious shell commands, the attacker achieves Remote Code Execution (RCE) and spawns a reverse shell, taking full control of the Ground Station pod.

**2. Lateral Movement (Log4Shell)**
With control over the edge, the attacker alters the telemetry requested by the Control Room. The Logstash instance in the Control Room uses a vulnerable version of Apache Log4j to process incoming data. The attacker spins up a rogue LDAP server on the compromised edge node and injects a malicious JNDI lookup string (`${jndi:ldap://...}`) into the telemetry payload. When Logstash parses this, it fetches and executes a malicious Java class, granting the attacker a second reverse shell—this time inside the central Control Room pod.

**3. Host OS Takeover (Container Evasion)**
The final hurdle is escaping the container. The Logstash deployment suffers from a severe, but surprisingly common, Kubernetes misconfiguration: its ServiceAccount is overly permissive, granting it the right to create new pods via the core K8s API.
    
Using the compromised pod's JWT token, the attacker interacts with the Kubernetes API to spin up a new, highly privileged pod with `hostPID` enabled and security boundaries disabled. By executing the `nsenter` command inside this new pod, the attacker steps out of the containerized namespaces entirely, landing a root shell directly on the Host OS.

### The Takeaway
This exercise highlights a critical lesson for niche operational technology (OT) fields like aerospace: adopting modern IT orchestration like Kubernetes brings immense scalability and resilience, but it also opens the door to standard IT kill chains. Proper Threat modelling, role-based access control (RBAC), strict network policies, and prompt patching are non-negotiable, whether your server is sitting in a traditional data center or communicating directly with satellites.
