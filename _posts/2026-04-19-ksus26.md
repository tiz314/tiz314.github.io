## New Era (Part 1)
A new era began: the trendy analog communication has a new rival. Are you prepared?

*Note: 8 samples per symbol!*

### Solution
The challenge provides a `.iq` file with no metadata other than an "8-sample-per-symbol" rate. In the SDR (Software Defined Radio) world, `.iq` almost always implies a raw binary stream of IQ (In-phase and Quadrature) data. 

To grasp some information about the data type, let's inspect the raw bytes:
```bash
hexdump -C intercepted_signal.iq | head -n 5
```
```text
00000000  00 00 80 bf 00 00 00 00  00 00 80 bf 00 00 00 00  |................|
*
00000040  00 00 80 3f 00 00 00 00  00 00 80 3f 00 00 00 00  |...?.......?....|
*
00000080  00 00 80 bf 00 00 00 00  00 00 80 bf 00 00 00 00  |................|
```

We can notice how the non-zero 32-bit blocks are either `00 00 80 3f` or `00 00 80 bf`. By checking online for a possible encoding of these values, we find references to the IEEE 754 floating-point standard, which represents values using 32 bits. Because the file is little-endian, these bytes translate to `0x3f800000` and `0xbf800000`. 

Using an online converter, we can discover that those two hex values exactly map to the floats **1.0** and **-1.0**. After each non-zero 32-bit float, an empty 32-bit float (`00 00 00 00`) is present. Given the `.iq` format, the values must be interpreted as pairs. This leads to alternating pairs of (non-zero, zero) values, which perfectly map to (In-phase, Quadrature) data points. 

For visual confirmation, Audacity can be used by importing the raw data. To view it correctly, we select the **32-bit float** encoding. 

Looking at the standard waveform view, it is possible to visually confirm that the signal is a digital baseband transmission with two different amplitudes mapped to binary 1 or 0. This is the exact behavior of the 2-PAM (Pulse Amplitude Modulation) technique.

We can now build a Python script using NumPy to read the floating-point pairs from the file, average the noise, and extract the bits. 

```python
import numpy as np

FILE_NAME = "intercepted_signal.iq"
SAMPLES_PER_SYMBOL = 8              # From challenge description

# Read pairs of 32-bit floats
iq_data = np.fromfile(FILE_NAME, dtype=np.complex64)    

# We only need the first, non-zero component (In-phase)
i_channel = iq_data.real                                

recovered_bits = []

# Process the signal in chunks of SAMPLES_PER_SYMBOL
for i in range(0, len(i_channel), SAMPLES_PER_SYMBOL):
 chunk = i_channel[i : i + SAMPLES_PER_SYMBOL]
    
    # Calculate the continuous average to filter out noise
 avg_val = np.mean(chunk)
    
    # Apply threshold to the averaged chunk to determine the bit
    if avg_val > 0:                         
 recovered_bits.append(1)
    else:
 recovered_bits.append(0)

# Assuming no preamble or padding, data starts at index 0
start_idx = 0
payload_bits = recovered_bits[start_idx:]

chars = []
# Group bits into bytes and convert to ASCII characters
for i in range(0, len(payload_bits), 8):
 byte_bits = payload_bits[i : i + 8]
    
    # Stop if we hit an incomplete byte at the end of the file
    if len(byte_bits) < 8:
        break
        
 bit_string = "".join(str(b) for b in byte_bits)
 byte_val = int(bit_string, 2)
 chars.append(chr(byte_val))

result = "".join(chars)
print(f"\nFlag: {result}")
```

## New Era (Part 2)
The analog loyalists tried to corrupt our new system with noise, so the engineers pushed a hasty firmware update.

We managed to exfiltrate their compiled `encoder` binary alongside the newest RF capture. Can you reverse their encoding scheme and recover the message? 

*Note: The physical transmission parameters (8 samples per symbol) remain exactly the same as before.*

### Solution

Since the physical layer parameters haven't changed from the previous challenge, we can reuse our 2-PAM demodulator script to extract the raw transmitted bitstream. We read the `.iq` file, average every 8 samples, apply our threshold, and save the raw bits to a binary file for further analysis.

```python
import numpy as np

FILE_NAME = "intercepted_signal.iq"
OUT_FILE = "demodulated_payload.bin"
SAMPLES_PER_SYMBOL = 8

# Load data and extract the I-channel
iq_data = np.fromfile(FILE_NAME, dtype=np.complex64)
i_channel = iq_data.real

recovered_bits = []

# Demodulate 2-PAM
for i in range(0, len(i_channel), SAMPLES_PER_SYMBOL):
 chunk = i_channel[i : i + SAMPLES_PER_SYMBOL]
 avg_val = np.mean(chunk)
 recovered_bits.append(1 if avg_val > 0 else 0)

# Save the raw bits (as 0x00 and 0x01 bytes) to a file
np.array(recovered_bits, dtype=np.uint8).tofile(OUT_FILE)
```

If we try to convert this directly to ASCII like we did last time, we get garbage. The challenge description hints at an encoding, and we are given an `encoder` ELF binary to investigate.

Loading the `encoder` binary into a decompiler like Ghidra, we can find the core encoding function. The decompiled pseudo-code looks something like this:

```c
void encode_nibble(byte param_1, byte *param_2) {
 byte bVar1 = param_1 >> 3 & 1;
 byte bVar2 = param_1 >> 2 & 1;
 byte bVar3 = param_1 >> 1 & 1;
 byte bVar4 = param_1 & 1;
  
  *param_2 = bVar1 ^ bVar2 ^ bVar4;
  param_2[1] = bVar1 ^ bVar3 ^ bVar4;
  param_2[2] = bVar1;
  param_2[3] = bVar2 ^ bVar3 ^ bVar4;
  param_2[4] = bVar2;
  param_2[5] = bVar3;
  param_2[6] = bVar4;
  return;
}
```

By analyzing the bitwise operations, we can deduce exactly what this algorithm is doing:
1. It takes a 4-bit input (`param_1` masked into `bVar1` through `bVar4`).
2. It generates three parity bits using XOR logic.
3. It outputs a 7-bit block (`param_2[0]` through `param_2[6]`).

This 4-to-7 mapping with XOR parity generation is the exact mathematical signature of a **Hamming(7,4)** block code. The layout of the output array is `[p1, p2, d1, p3, d2, d3, d4]`.

To recover the flag, we need to reverse this process. We can write a script (in Python or C) that reads the demodulated bits in chunks of 14 (two 7-bit blocks per ASCII character). For each 7-bit block, we calculate the error syndrome to fix any bit flips introduced by channel noise, and then extract the 4 original data bits.

Here is the C implementation of the decoder:

```c
#include <stdio.h>
#include <stdint.h>

// Decodes a 7-bit Hamming block and returns the corrected 4-bit nibble
uint8_t decode_nibble(uint8_t *r) {
    // Calculate syndrome bits based on the parity logic we reversed
    uint8_t s1 = r[0] ^ r[2] ^ r[4] ^ r[6];
    uint8_t s2 = r[1] ^ r[2] ^ r[5] ^ r[6];
    uint8_t s3 = r[3] ^ r[4] ^ r[5] ^ r[6];
    
    // Syndrome value gives the 1-based error position
    uint8_t error_pos = s1 * 1 + s2 * 2 + s3 * 4;
    
    // Correct the flipped bit if an error was detected
    if (error_pos != 0) {
        r[error_pos - 1] ^= 1;
 }
    
    // Extract the corrected data bits (d1, d2, d3, d4)
    return (r[2] << 3) | (r[4] << 2) | (r[5] << 1) | r[6];
}

int main() {
 FILE *f = fopen("demodulated_payload.bin", "rb");
    if (!f) return 1;
    
    uint8_t buffer[14]; // Buffer for two 7-bit blocks (1 char)
    
    printf("Flag: ");
    while (fread(buffer, 1, 14, f) == 14) {
        // Decode the high nibble (first 7 bits) and low nibble (next 7 bits)
        uint8_t high_nibble = decode_nibble(buffer);
        uint8_t low_nibble = decode_nibble(buffer + 7);
        
        // Combine into the final ASCII character
        char c = (high_nibble << 4) | low_nibble;
        printf("%c", c);
 }
    printf("\n");
    fclose(f);
    return 0;
}
```

Compiling and running the decoder on our `demodulated_payload.bin` yields the flag.

## Ring 0 Security (?)
We've intercepted the driver for a proprietary decryption device. It appears that a 4-digit PIN or similar is required, but we are unable to determine what it is. Can you help us?

Note: the module was compiled for Linux 6.12.63. A matching QEMU environment is already provided :)

### Solution
The compiled kernel object is attached to the challenge. By loading it, a device at `/dev/decoder` is created. 

We can check the binary using Ghidra to discover the custom functions supported by the driver, which are custom `ctls` operations. The first one: 
```c
lVar4 = _copy_from_user(&uStack_14, param_3, 4);
if (lVar4 == 0) {
    session_key[1] = 0xcafebabe;
    session_key[0] = uStack_14 | 0x13370000;
    session_key[2] = 0xdeadbeef;
    session_key[3] = 0xfeedface;
    _printk(&DAT_00100620);
 uVar3 = 0;
}
```
takes the user input parameter and updates the first byte of the `session_key` variable by ORing it to the initial value. 

In the second custom operation, the driver uses the current `session_key` content to perform cryptographic operations, whose output is then returned to user space. 

We can write a C program to open the device and send `ioctl` commands using the command IDs we can extract from the decompiled code: 

```c
    int fd = open("/dev/decoder", O_RDWR);

    uint32_t pin = 0000;
    ioctl(fd, 0x401b3700, &pin);

    char res[128];
    ioctl(fd, 0x801b3701, res);
```

Since we don't know the correct pin, but we do know that the start of the flag is KSUS, we can perform a known plaintext attack without even knowing how the cryptographic algorithm works. We do know from the decompiled code that the user PIN is a `uint32_t`, so 4 bytes. We also know from the challenge description that we are looking for a 4-digit PIN. This heavily reduced the searching space, and we can easily brute force it via scripting all the possible combinations from `0000` to `9999`. 

In this way, we'll stop at a given PIN in this range since the first characters of the result coming from the driver are the `KSUS` characters. 

`solver.c`:
```c
#include <stdio.h>
#include <fcntl.h>
#include <unistd.h>
#include <stdlib.h>
#include <sys/ioctl.h>
#include <stdint.h>
#include <string.h>

// Check if a string looks like a flag
int is_valid_flag(const char *str) {
    if (strstr(str, "KSUS") != NULL) {
        return 1;
 }
    return 0;
}

int main(void){
    int fd = open("/dev/decoder", O_RDWR);
    char res[128];
    uint32_t pin;

    // PIN bruteforce
    for (pin = 0; pin <= 9999; pin++) {
        
        // Send the current PIN guess
        if (ioctl(fd, 0x401b3700, &pin) < 0) {
            perror("Error setting key");
            close(fd);
            return -1;
 }

        memset(res, 0, sizeof(res));
        if (ioctl(fd, 0x801b3701, res) < 0) {
            perror("Error reading flag");
            close(fd);
            return -1;
 }

        if (is_valid_flag(res)) {
            printf("PIN: %d\n", pin);
            printf("FLAG: %s\n", res);
            close(fd);
            return 0;
 }
 }
    close(fd);
    return 0;
}
```
